# Roguelike_Ship ‚Äî Unity 2D Ship Roguelike

Top-down 2D ship roguelike built in Unity with URP 2D Render Pipeline. Features modular weapon systems, a stage progression bar, and mouse-aimed turret combat.

## Class Reference

### `GameManager.cs`
- **Path:** `Assets/Scripts/GameManager.cs`
- **Base:** `MonoBehaviour`
- **Purpose:** Top-level stage controller. Singleton. Drives the stage timer, holds the sorted encounter list, and triggers encounters as progress advances. Optionally freezes progress while an encounter is active (when `DistanceBar.waitForEncounterCompletion` is enabled).
- **Key Fields:**
  - `Instance` (`static GameManager`) ‚Äî singleton accessor
  - `_distanceBar` (`DistanceBar`) ‚Äî reference to the UI progress bar
  - `_encounterSpawner` (`EncounterSpawner`) ‚Äî spawns encounters on stage start
  - `encounters` (`List<Encounter>`) ‚Äî sorted list of active encounters
  - `totalStageDuration` (`float`) ‚Äî total length of the stage in seconds
  - `timeSinceStageStart` (`float`, private) ‚Äî elapsed time for current stage
  - `nextEncounterIndex` (`int`, private) ‚Äî cursor into the sorted encounter list
- **Key Methods:**
  - `Awake()` ‚Äî sets singleton Instance
  - `Start()` ‚Äî initialises timer, sorts encounters, calls `_encounterSpawner.GenerateStageEncounters()`
  - `Update()` ‚Äî increments timer by `Œîtime √ó timeScale`, pushes progress to bar, checks encounters; skips time increment when `waitForEncounterCompletion` is true and an encounter is active
  - `RegisterEncounter(Encounter)` ‚Äî adds encounter to list, re-sorts by triggerPoint
  - `CheckEncounters()` ‚Äî fires next encounter when its triggerPoint is reached, clamps `timeSinceStageStart` to exact trigger point if `waitForEncounterCompletion` is enabled, advances cursor
  - `HasActiveEncounter()` ‚Äî returns true if any encounter has `IsActive == true`
- **Relationships:** Depends on `DistanceBar`, `EncounterSpawner`, `Encounter`, `TimeController`

### `TimeController.cs`
- **Path:** `Assets/Scripts/TimeController.cs`
- **Base:** `MonoBehaviour`
- **Purpose:** Manages time state (Paused/Normal/Fast) via `Time.timeScale`. Singleton with UI button and keyboard (Space) controls.
- **Key Fields:**
  - `Instance` (`static TimeController`) ‚Äî singleton accessor
  - `pausedScale`, `normalScale`, `fastScale` (`float`) ‚Äî time scale for each state
  - `pauseToggleKey` (`KeyCode`) ‚Äî key to toggle pause (default Space)
  - `CurrentState` (`TimeState`) ‚Äî current time state (Paused/Normal/Fast)
  - `CurrentTimeScale` (`static float`) ‚Äî convenience accessor for `Time.timeScale`
- **Key Methods:**
  - `Pause()`, `SetNormal()`, `SetFast()` ‚Äî set state directly
  - `TogglePause()` ‚Äî toggles between Paused and Normal
  - `Update()` ‚Äî handles Space key input
- **Events:**
  - `OnTimeStateChanged` ‚Äî fired when time state changes
- **Relationships:** Used by `TimeControlUI`, `GameManager`, `GatlingGunModule`

### `TimeControlUI.cs`
- **Path:** `Assets/Scripts/TimeControlUI.cs`

- **Key Fields:**

### `Encounter.cs`
- **Path:** `Assets/Scripts/Encounter.cs`
- **Base:** `MonoBehaviour`
- **Purpose:** Abstract base class for all encounter types. Each has a trigger point along the distance bar and a HUD marker icon.
- **Key Fields:**
  - `triggerPoint` (`float`) ‚Äî progress [0,1] at which this encounter activates
  - `_hudIconTexture` (`Texture2D`) ‚Äî custom texture for the HUD marker (optional, auto-generates a white circle if null)
  - `_hudIconImage` (`Image`, private get) ‚Äî runtime Image component assigned by DistanceBar
- **Key Properties:**
  - `IsActive` (`bool`) ‚Äî true when the encounter has been fired but not yet completed
- **Key Methods:**

  - `SetIconPosition(Vector2)` ‚Äî positions the HUD icon

- **Path:** `Assets/Scripts/Player.cs`

- **Purpose:** Central player controller. Singleton. Manages health, collision damage, death, and the ship's power budget. Attached to `PlayerShip` GameObject.
- **Key Fields:**

  - `currentAmmo` (`int`, private) ‚Äî rounds remaining in current magazine

## Scene Summary ‚Äî `SampleScene.unity`

The main (and only) scene. Contains:

- **GameManager** GameObject ‚Äî hosts `GameManager.cs`, `TimeController`, `DistanceBar` components
  - **Encounters** child ‚Äî hosts `EncounterSpawner` component
- **PlayerShip** ‚Äî root GameObject for the player. Has no SpriteRenderer; visuals come from child modules. Components: `Rigidbody2D`, `CompositeCollider2D`, `Player.cs`

Encounters are spawned programmatically by `EncounterSpawner` at stage start. HUD markers for each encounter are created at runtime by `DistanceBar.PositionEncounterMarker()`.

## Known Gaps

- **`ShipModule.cs`** ‚Äî minimal base class with power fields only; no virtual methods, no shared module interface beyond power.
- **No enemy/AI system** ‚Äî no spawner, no enemy behaviour, no collision/damage.
- **No shields/engines/sensors systems** ‚Äî only weapons (GatlingGunModule) wired up; other system types exist as stubs in the design doc but have no modules yet.
- **No warning system** ‚Äî no visual/audio cue before encounter triggers.
- **No boss fights** ‚Äî no boss encounter type, no stage completion logic.
- **No roguelike upgrades** ‚Äî no mid-run upgrade choices after encounters.
- **`DistanceBar.updateVisual()`** ‚Äî position math may contain off-by-one issues (references `_blankBackBar.rect.height` with hardcoded `-1.5f` offset).
- **`GameManager`** has no stage lifecycle ‚Äî stage runs immediately, no start/end events, no win/lose condition.
- **No Input Action Map integration** ‚Äî `InputSystem_Actions.inputactions` asset exists but scripts use legacy `Input.mousePosition` / `Input.GetMouseButton()` directly.

## Coding Practices

### Naming Conventions

Public fields use underscore prefix (`_`) when they are **object/reference assignments** ‚Äî GameObjects, Components, Transforms, or other asset references that get dragged into Inspector slots (e.g. `_turret`, `_bulletSpawnPoint`, `_distanceBar`, `_blankBackBar`).

Public fields that are **value sliders or numeric tunables** do not get the underscore prefix (e.g. `rotationSpeed`, `fireRate`, `maxHealth`, `invincibilityTime`).

This makes it immediately clear in the Inspector which fields are wiring references vs. tuning values.

### Public fields in scripts are defaults, not authoritative values

Public fields on `MonoBehaviour` scripts (e.g. `maxHealth`, `count`, `spawnArea`) set **default values only**. The actual values used at runtime come from the **scene** (for GameObjects placed in the world) or **prefab** (for prefab instances). These serialized values should be tuned in the Unity Editor, not hardcoded in `.cs` files.

- **Do** set sensible defaults in code that make a script work out of the box.
- **Do** adjust values in the Inspector / prefab override for final tuning.
- **Do not** change public property defaults in `.cs` files as a substitute for editing scene/prefab serialized values.
- **Exception:** Trivial test scripts that are never tuned in the Editor.
